## 分析

### v3で改善された点
- Action パターンを18個に拡充
- Noun patterns を6種類に拡張（body_part, object, location, visual, frame, temporal）
- スコアリング方式で複数パターンナットの競合を改善
- → Action精度が44% → 62% に向上

### v3の残る課題

#### 1. Action（62%）の失敗パターン
- **"dolly_in"が保存される**: 多くの camera motion が誤判定
  - 例: "Apply a smooth dolly in..." → preserve_framing
  - 原因: 複数パターン（"smooth", "apply"など）が競合
  
- **"add_object"が検出されない**: 
  - 例: "Increase animals by adding..." → apply_style
  - 原因: "増加"を意味する動詞の複雑な構文

- **長文instruction での誤分類**
  - パターン競合が複合

#### 2. Target（28%）の失敗
- **汎用パターンの限界**
  - "body_part" パターンが強すぎて、他のターゲットを誤検出
  - noun_patterns は6種類だが、GT には実に多様な target がある

---

## 根本的な問題

単純な**正規表現パターンマッチング**には以下の限界がある：

1. **Semantic 理解の欠如**
   - instruction は自然言語で、構文が多様
   - パターンでは「意味」を捉えられない

2. **Target の多様性**
   - GT に100種類以上の異なる target が存在
   - 汎用パターン6種では不足

3. **複雑な instruction の処理**
   - 複合文、修飾句、複数タスク
   - 単純な正規表現では対応困難

4. **Action パターンの競合**
   - 複数パターンが同時にマッチ
   - スコアリングだけでは解決不十分

## 設計の選択肢

### オプション A: 現在のパターン方式の改善
```
実装コスト: 低
期待精度: 70～75%（予想）
方法:
  1. action_patterns をさらに詳しく（現在16 → 30+）
  2. noun_patterns を詳細化（現在6 → 20+）
  3. ファジーマッチング導入

課題: 細かなチューニングの繰り返し、ドメイン知識が必要
```

### オプション B: 軽量な Deep Learning（推奨）
```
実装コスト: 中
期待精度: 80～90%
方法:
  1. BERTカディング器 + simple classifier
  2. action: sentence-BERT ( instruction) → action embeddings → action分類
  3. target: Named Entity Recognition (NER) × instruction

利点: 
  - semantic 理解が可能
  - 一度学習すれば汎用性高い
  - チート情報なし設計を保持
```

### オプション C: 大規模言語モデル（最小コスト）
```
実装コスト: 非常に低
期待精度: 85～95%
方法:
  1. OpenAI GPT / Local LLM に few-shot examples で指示
  2. instruction → {action, target} の直接出力

利点:
  - 最小実装時間
  - 高精度
  - Semantic 理解完璧

課題: API コスト、レイテンシー、ローカル実行時はメモリ
```
