# submit_baseline_ver06 メモ

## 1. `dolly_in ---> pass throught` になる理由

`stable_zoom_in()` 自体は定義されていますが、現在の入力ではその分岐まで到達していません。

### 確認できた挙動

- rule JSON 上で `dolly_in` は `progressive_crop_resize` に正しくマッピングされています。
- 対象 GT の先頭 task（`dolly_in`）には `constraints: ["smooth_motion"]` が含まれています。
- `run_method()` では、method 分岐より先に constraint ベースの passthrough 判定が実行されます。
- `"smooth_motion" in constraints` が真のため、その時点で `frames` を早期 return します。
- その結果、後段の `if method == "progressive_crop_resize": return stable_zoom_in(...)` は実行されません。

### 直接原因

現在の制御フローでは、method 実行よりも constraint による passthrough ゲートが優先されています。
そのため `smooth_motion` を含む task は、実処理メソッドを通らずにスキップされます。


## 2. この設計になっている意図（推定）

現状の実装は、保守的なベースライン方針を反映した設計である可能性が高いです。

### 想定される意図

- constraints を品質・安全要件として強く扱う。
- その要件を安定して満たせない実装段階では、無理に編集せず passthrough を選ぶ。
- 開発初期に flicker / jitter / artifact を増やさないことを優先する。

### ベースラインとしての妥当性

- 信頼性が不十分なときは、低品質な編集より no-op を優先できる。
- 一部オペレータ未完成でも、全体パイプラインを壊しにくい。
- 「動いているように見えるが品質が悪い」状態を避けられる。

### トレードオフ

- 利点: 品質要件が厳しい条件で安全側に倒せる。
- 欠点: `dolly_in` のように method が実装済みでも実行されない場合がある。

### 運用メモ

`stable_zoom_in` などの camera motion 処理を実際に通したい場合は、
method 分岐を constraint passthrough 判定より前に置く、
または constraint を「ハードスキップ条件」ではなく
「事後評価・補助ヒント」として扱う設計に変更するのが有効です。


## 3. 今回の変更内容（ver06）

要望に合わせて、`constraints` による分岐を現状すべて無視するように変更しました。

### 変更ファイル

- [src/postprocess/task_rules_ver05_functions.py](src/postprocess/task_rules_ver05_functions.py)

### 変更点

- `run_method()` 内の constraint ベース passthrough 分岐
	（`unchanged`, `smooth_motion`, `no_jitter` など）を無効化。
- これにより、constraint の有無に関係なく method 分岐
	（例: `progressive_crop_resize` -> `stable_zoom_in`）が実行されます。

### 影響

- これまで `constraints` で止まっていた camera motion 系タスクが実行されるようになります。
- 一方で、制約を品質ゲートとして使う挙動は一時的に失われるため、
	品質面の検証（flicker, jitter, artifacts）は別途確認が必要です。


## 4. dolly_in の修正内容

`dolly_in` については、従来の「画面全体を crop + resize する疑似ズーム」ではなく、
「対象オブジェクト全体だけを徐々に大きく見せる」方式へ変更しました。

### 修正背景

- 従来の `stable_zoom_in()` は、対象中心で画面全体を切り出して拡大する実装でした。
- そのため、背景も一緒に拡大される camera zoom 的な挙動になっていました。
- 今回の要件では、`dolly_in` で背景はそのままにしつつ、対象オブジェクト全体だけを大きくしたい状態でした。
- また、`man's face` のような target では、`face` だけではなく `man` を含む対象全体を捉える必要がありました。

### 修正意図

- `target` をそのまま1つの prompt として使うのではなく、単語ごとに分割して GroundingDINO に渡す。
- 例えば `man's face` であれば、`man` と `face` を個別に検出し、得られた bbox の結合領域を使う。
- その結合 bbox を SAM に渡して mask を取得することで、対象全体を含む領域を扱えるようにする。
- `dolly_in` 時は、その mask 領域だけを元フレームから切り出して拡大し、背景はなるべく変更しない。

### 実装内容

- `target` 文字列を分割する補助関数を追加。
- 分割したキーワードごとに GroundingDINO を実行し、bbox を結合する処理を追加。
- 結合 bbox を SAM に渡して対象 mask を取得する処理を追加。
- `dolly_in` / `motion_type == dolly_in` の場合は、専用の object-only zoom 経路へ入るよう変更。
- 拡大時は mask 部分だけを合成し、背景は元フレームを保持する。

### 結果

- `dolly_in` で、対象オブジェクト全体が時間とともに大きく見える挙動へ変更。
- `man's face` のような target でも、`face` 単独ではなく `man + face` の結合領域ベースで処理される。
- 背景全体を一緒にズームする従来挙動より、要件に近い見え方になる想定。

### 注意点

- 現状は初回フレームで決めた結合 bbox / mask を基準に拡大しているため、対象の大きな移動にはまだ強くありません。
- より追従性を高める場合は、フレームごとの bbox 更新や mask 更新を別途入れる余地があります。


## 5. dolly_in の追加試行錯誤

上記の object-only zoom 化だけでは、見た目としてまだ要件を満たしていませんでした。
そのため、以下の順で修正を積み増しました。

### 5.1 bbox 拡大では背景も一緒に拡大される問題

最初の段階では、対象 bbox を基準に拡大したパッチを元フレームへ重ねていました。
この方法だと、bbox 内に含まれる背景も一緒に拡大されるため、
「人物だけが大きくなる」のではなく「人物周辺の長方形領域全体が大きくなる」状態でした。

この段階で、bbox ベースではなく SAM の mask 領域だけを拡大対象にする方針へ変更しました。

### 5.2 初回フレーム固定 mask では追従できない問題

次に、初回フレームで作った mask を全フレームで使い回す実装では、
対象の位置や姿勢が変わったときにずれが出ました。

そのため、`dolly_in` 専用経路では各フレームごとに以下を毎回実行するよう変更しました。

- GroundingDINO で target に対応する bbox 群を取得
- それらの結合 bbox を算出
- SAM でそのフレームの mask を再生成
- その mask 領域だけを拡大して合成

これにより、対象の移動に対して毎フレーム追従する形になりました。

### 5.3 上側が切れないようにする調整

mask を拡大すると、対象の上端がフレーム外へはみ出して頭部が切れるケースがありました。
今回の要件では、下側は多少切れてもよい一方で、上側は切れない方が望ましい状態でした。

そのため、mask 拡大時の affine 変換に補正を入れ、
拡大後の mask 上端が `y=0` より上へ出る場合は、そのぶんだけ下方向へ平行移動するようにしました。

これにより、上側の欠けを抑えつつ、下側のクリップは許容する挙動に調整しています。

### 5.4 元の人物が背景に残る問題

さらに、mask 部分だけを拡大して重ねるだけでは、
拡大前の人物が元フレーム側に残り続ける問題がありました。

つまり、前景として大きくした人物は描かれるものの、
背景レイヤーには元の人物がそのまま残ってしまい、残像のように見える状態でした。

この問題に対しては、合成前に元の mask 領域を背景から除去し、
近傍画素で補完したうえで、その上へ拡大後の人物だけを貼るように変更しました。

具体的には以下の処理です。

- 元 mask を少しだけ膨張させる
- `cv2.inpaint(..., cv2.INPAINT_TELEA)` で周辺色から背景を補完する
- 補完済み背景を土台にして、拡大後の mask 前景だけを重ねる

この変更により、拡大前の人物が背景に残る問題は解消しました。

### 5.5 最終的な到達点

最終的な `dolly_in` は、以下の流れになっています。

- 各フレームで GroundingDINO + SAM により対象 mask を再生成する
- mask 領域だけを拡大する
- 拡大前の mask 領域は inpaint で背景補完して消す
- 拡大後の上端が切れそうな場合は下方向へ補正する

今回の確認では、`check_submission_ver06.ipynb` 上で再実行し、
少なくとも「元の人物が背景に残る」問題は解消できたことを確認した。

### 5.6 今後の改善余地

- フレームごとの mask 再推定により、境界がわずかに揺れる可能性がある
- inpaint の補完結果は背景テクスチャによっては不自然になる可能性がある
- 必要であれば、mask 境界の feathering や時系列平滑化を追加する余地がある
